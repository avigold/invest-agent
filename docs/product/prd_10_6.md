# PRD 10.6 — Recommendation Change Alerts

**Status**: Draft
**Feature**: #7 from Bloomberg/YCharts parity roadmap
**Depends on**: Existing scoring systems (deterministic + ML)

## Problem

Users have no way to know when a company's Buy/Hold/Sell classification changes. Both scoring systems compute classifications on the fly from stored scores — they are never persisted. When a company flips from Buy to Hold after a re-scoring run, nobody notices unless they happen to be looking at that company's page. Professional terminals surface signal changes automatically.

## Solution

Track classification changes via a snapshot-and-diff mechanism at scoring job boundaries. A new `signal_changes` table records each flip. An API endpoint exposes recent changes. The Dashboard surfaces them in a compact card.

## How It Works

1. Before a scoring job begins, snapshot current classifications for all companies
2. After scoring completes and commits, recompute classifications
3. Diff the two snapshots — any classification that changed gets logged to `signal_changes`
4. The Dashboard queries recent changes and renders them as a card

## Data Model

### New Table: `signal_changes`

| Column | Type | Notes |
|--------|------|-------|
| `id` | UUID | Primary key |
| `ticker` | String(20) | Denormalised, no FK |
| `company_name` | String(255) | Denormalised |
| `system` | String(20) | `"deterministic"` or `"ml"` |
| `old_classification` | String(10) | Buy, Hold, or Sell |
| `new_classification` | String(10) | Buy, Hold, or Sell |
| `old_score` | Float | Composite score before |
| `new_score` | Float | Composite score after |
| `detected_at` | DateTime(tz) | When the change was detected |

Indexes: `detected_at` (for recency queries).

No foreign keys — intentionally denormalised to avoid cascade complexity and to remain readable even if the originating score rows are later deleted.

## API

### `GET /v1/signals/changes`

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `limit` | int | 20 | Max rows (1–100) |
| `system` | string? | null | Filter by `deterministic` or `ml` |

Returns `SignalChange[]` ordered by `detected_at DESC`.

## Hook Points

### ML System (`prediction_score_handler`)

Before deleting old `PredictionScore` rows:
- Load all scores for the model
- Compute classification + composite for each via `score_from_features()` + country/industry lookups
- Store as `{ticker: (classification, composite)}`

After inserting new scores and committing:
- Repeat the same computation
- Diff with the old snapshot
- Insert `SignalChange` rows for any flips

### Deterministic System (`score_sync_handler`)

Before the first scoring batch:
- Call `compute_recommendations(db)` to get all current classifications
- Store as `{ticker: (classification, composite_score)}`

After the final commit:
- Call `compute_recommendations(db)` again
- Diff with the old snapshot
- Insert `SignalChange` rows for any flips

### First-run handling

If the old snapshot is empty (no existing scores), skip change detection silently. No false positives.

## UI Design

### Dashboard Card

Compact card in the ML section titled "Signal Changes":
- Each row: ticker (link), old → new classification with arrow, system pill, relative timestamp
- Colour: upgrades (toward Buy) in green, downgrades (toward Sell) in red
- Empty state: "No recent signal changes"
- Max 10 items shown
